**Прежде чем писать код:**

## 🔍 ОБЯЗАТЕЛЬНО: Найди ВСЕ связанные файлы сам

**Пользователь может забыть указать файл. ИИ обязан САМ найти все связанные файлы:**

1. **Сначала** прочитай файлы, которые указал пользователь
2. **Затем** через `grep_search` найди все упоминания классов/методов/функций
3. **Затем** через `glob` найди все файлы в затронутых папках
4. **Прочитай** каждый найденный файл перед изменением

**НЕ начинай писать код, пока не нашёл и не прочитал ВСЕ связанные файлы.**

## 🎨 Правила работы с UI/Layout (если применимо)

### Grid / Table Layout
- **Всегда считай колонки**: если элемент на колонке 9, должно быть минимум 10 определений
- **После добавления элемента**: проверь что Grid/Table имеет достаточно колонок/строк
- **Никогда не добавляй элемент без проверки** определений

### Flex / Stack / Wrap контейнеры
- Горизонтальный стек может выйти за экран — используй scroll если элементов много
- Для элементов в ряд: предпочитай Grid с авто-шириной колонок

### Проверка перед изменением UI
1. Прочитай весь контейнер целиком
2. Посчитай определения колонок/строк
3. Проверь каждый элемент на перекрытие
4. Убедись, что отступы корректны

### После изменения UI
1. Перечитай весь файл
2. Проверь что все элементы на своих местах
3. Убедись, что элементы не накладываются друг на друга

## 🎯 Как понять, что от тебя хотят

### Пользователь написал задачу:
```
1. Пойми суть задачи (если непонятно — спроси)
2. Найди связанные файлы в AI/tasks.md
3. Прочитай соответствующие файлы кода
4. Составь план (покажи пользователю)
5. Реализуй по шагам
6. Проверь себя
7. Расскажи что сделал
```

### Если пользователь просит "починить баг":
1. Прочитай описание бага в `AI/tasks.md`
2. Найди упомянутые файлы
3. Пойми корень проблемы
4. Если не уверен — **спроси пользователя**
5. Исправь минимальным изменением
6. Проверь, что не сломал другое

### Если пользователь просит "добавить фичу":
1. Уточни детали (если нужно)
2. Спроектируй минимальное решение
3. Покажи план
4. Реализуй пошагово
5. Обнови `AI/tasks.md`
